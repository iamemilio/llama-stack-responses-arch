Llama Stack's Responses API implements the OpenAI Responses API contract on top of configurable inference providers. It adds an agentic tool loop, conversation state, guardrails, and persistent storage — so you get the same client interface without vendor lock-in.

API & compatibility
OpenAI wire format
Same POST/GET/DELETE /v1/responses endpoints, request models, and SSE event sequence. Existing OpenAI SDK clients work without changes. This means teams already using the OpenAI Responses API can point their client at a Llama Stack server and keep their application code as-is — no new SDK, no rewrite.
Pluggable inference
Inference is routed through whichever backend is configured — OpenAI, vLLM, Ollama, Bedrock, Azure, WatsonX. The Responses API surface stays the same. This decouples your application from any single model provider, so you can evaluate models side-by-side, run locally in development, or switch providers in production without touching client code.
Tool execution
Agentic tool loop
Runs inference → tool call → inference cycles automatically, up to a configurable max_infer_iters. Supports parallel tool calls. The server handles the orchestration loop so clients don't need to implement their own retry/dispatch logic — you send a single request and get back a fully resolved response, regardless of how many tool calls were needed.
Built-in tools
web_search and file_search (vector store RAG with ranking, filters, and source annotations) ship out of the box. file_search is particularly useful because it integrates directly with the Vector IO API — you can point it at your vector stores and get grounded, citation-backed answers without building a separate retrieval pipeline.
MCP connectors
Connects to remote MCP tool servers. Sessions are reused within a request and restored across turns via previous_response_id. This lets you integrate with the growing MCP ecosystem — databases, APIs, code execution environments — by just pointing at a server URL rather than writing custom tool code.
Custom tools
Anything not built-in goes through the Tool Runtime API, so you can register your own tool groups. This gives you an escape hatch for domain-specific logic (internal APIs, proprietary systems) while keeping it inside the same agentic loop.
Conversation & state
Response chaining
previous_response_id loads a prior response's input + output as context. Tool and MCP state carry over automatically. This is the simplest way to build multi-turn agents — the client just passes an ID, and the server reconstructs the full context window including any tool results from earlier turns.
Managed conversations
conversation threads provide server-managed multi-turn history via the Conversations API. Useful when you need the server to own the conversation lifecycle — for example, when multiple clients or services contribute to the same thread, or when you want to inspect/modify conversation state independently of individual responses.
Persistent storage
SQL-backed store with access control. Responses are upserted incrementally during streaming, so GET works mid-stream. Full CRUD for list, retrieve, delete. This matters for observability and debugging — you can inspect any response after the fact, see exactly what the model received, and build audit trails without additional infrastructure.
Streaming & execution
SSE streaming
OpenAI-aligned event sequence (response.created, deltas, tool events, response.completed). Non-streaming mode consumes the same internal pipeline. Streaming is especially valuable for tool-heavy responses — users see progress in real time as each tool executes, rather than waiting for the full loop to complete.
Background jobs
background: true returns immediately with status: "queued". Results are polled via GET. Bounded worker queue with back-pressure. This is designed for workflows where the caller doesn't need to block — batch processing, async pipelines, or cases where a response might take minutes due to complex tool chains.
Safety & prompts
Guardrails
Input shields run before generation. Output shields run incrementally on accumulated streamed text. Violations yield structured refusals. Because guardrails are integrated at the orchestration layer, they apply consistently regardless of which inference provider or tools are in use — you define safety policy once and it covers all requests.
Prompt templates
Reference versioned prompts by ID. Variables are validated and substituted at runtime. Separate instructions field for system-level guidance. Centralizing prompts on the server means you can update them without redeploying clients, A/B test different versions, and enforce consistency across multiple applications hitting the same API.

Agent Loop
Core execution cycle inside StreamingResponseOrchestrator:
1. Guardrails
Run safety shields on input. Violations produce a refusal and exit.
2. Inference
Call Inference.openai_chat_completion(stream=True). Translate chat completion chunks into Responses stream events.
3. Tool Execution
If model returns tool calls, ToolExecutor dispatches them (web_search, file_search, MCP, user functions). Results appended to messages.
↺ Repeat steps 1–3 until no more tool calls
streaming.py, tool_executor.py

responses_store.py
Terminal Event
Emits response.completed, response.incomplete, or response.failed. Stream ends.
